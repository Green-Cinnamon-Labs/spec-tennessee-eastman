# Avaliação — o que o TEP expõe hoje via `simulation-framework` (base para `AcquisitionLayer`)

Levantamento objetivo do estado atual (`tep-plant`, branch `composite`, commit `9b0031b`) contra o papel de `AcquisitionLayer` definido em `plan_refactor.md` (seção 3). Conclusão geral: quase nada do que um gateway OPC-UA precisaria está de fato pronto hoje — o que existe é, majoritariamente, código legado já apagado (branch `main` / commit `603c0cd`, incluindo o antigo `service` crate com gRPC) que documenta a *intenção* do design, não o estado corrente. `XMEAS`, `XMV`, tempo de simulação, modo operacional e shutdown não existem no código ativo; o pipeline que os produziria (`Flows`, `Heat`, `Measurements`) está com `todo!()` no lugar do corpo. O que sobrevive de fato compilando e com dados reais são só os valores físicos brutos de `Reactor`/`Separator`/`Stripper`/`Compressor`.

Esses valores físicos (`reactor.temperature`, `reactor.pressure`, `reactor.liquid_volume`, `reactor.liquid_composition.{0..8}`, `reactor.vapor_composition.{0..8}`, `reactor.reaction_rates.{0..8}`, e o subconjunto menor equivalente em `separator.rs`/`stripper.rs`/`compressor.rs`) já são declarados como slots nomeados via `subscribe()` no `StateRegistry`, endereçáveis por `Proxy` — ou seja, o mecanismo de nomeação/resolução que uma `AcquisitionLayer` usaria já existe e funciona. Mas são campos privados dentro de cada struct (`temperature: Proxy`, não `pub`), sem getter público em `Reactor` nem em `TennesseeEastmanModel` — hoje só é possível lê-los abrindo o mesmo `Rc<RefCell<StateRegistry>>` usado na construção e fazendo `subscribe(&[], &["reactor.temperature", ...])` como um novo consumidor. Não existe nenhuma API de snapshot que devolva "todos os valores atuais nomeados" de uma vez.

`IDV` (distúrbios) é o único write path realmente implementado: `Disturbance::set_disturbances(&mut self, idv: &[i32; 20])` e `set_step_magnitude` (`disturbance.rs`) escrevem os 20 flags e alimentam 12 canais internos, com 4 saídas expostas como `Proxy` no registry. Só que `Disturbance` não é instanciado em `TennesseeEastmanModel::new` — é um componente órfão, sem efeito no modelo composto ativo. `XMV` está na mesma situação, um degrau pior: `Valve`/`Agitator` (em `simulation-framework/src/actuator/dynamic.rs`) já têm dinâmica de primeira ordem completa e `set_command()` funcional, mas nenhum dos dois é referenciado em lugar nenhum de `tep-plant` — nem instanciado, nem ligado a nada. `XMEAS` propriamente dito nunca existiu como campo nomeado em nenhuma versão do código, nem a antiga (sempre foi `Vec<f64>`/array cru indexado por posição) nem a atual (onde `Measurements::evaluate()` é literalmente `todo!(...)`, dependente de `Flows`/`Heat` que também são `todo!()`). Não há enum de modo operacional em nenhuma versão, e o flag de shutdown (existente no código antigo, calculado a partir de pressão/nível/temperatura — "Block 36" do `teprob.f`) desapareceu junto com `Measurements`. Controladores (havia um `PController`/`ControllerBank` com setpoint/bias observáveis) e qualquer snapshot/telemetria (havia `MetricsSnapshot` consumido por gRPC) também foram deletados por inteiro — só sobrevivem como código morto recuperável via `git show 603c0cd`.

`simulation-framework` já tem os blocos genéricos de sensor/atuador que uma camada de aquisição usaria para introduzir ruído/atraso realista (`Sensor`/`IdealSensor`/`FI`/`PI`/`LI`/`TI`/`AI` em `sensor/model.rs`, `FirstOrderSensor`/`SampledSensor` em `sensor/dynamic.rs`), mas nenhum é usado pelo TEP hoje — e os dois sensores dinâmicos têm o `impl DynamicModel` inteiro comentado, então nem compilam como componentes de fato. Isso é estado intencional, não regressão: `eval_kickoff.md` já registra que sensor/atuador são "passthrough sem uso real" e aponta as issues #56/#57 como o lugar certo para injetá-los de verdade e para construir a fachada de instrumentação (`ProcessImage`, com `NodeId`/nome/unidade/faixa/status por variável) — ou seja, o design-alvo já está escrito, só não implementado.

Dado esse estado, o conjunto mínimo que eu recomendaria expor primeiro, em ordem de esforço crescente, é: 
**(1)** um snapshot de leitura genérico sobre o `StateRegistry` — uma função que recebe uma lista de chaves e devolve os valores resolvidos, reaproveitando o `Proxy` que já existe, sem exigir nenhuma física nova; 
**(2)** um relógio de simulação — hoje inexistente até em `Simulation` (`simulation-framework/src/simulation.rs`), mas barato de adicionar e pré-requisito para qualquer sinal com timestamp; 
**(3)** o write path de `XMV`, religando `Valve`/`Agitator` (já implementados) em `TennesseeEastmanModel::new`, o que dá comando + posição observável de graça; 
**(4)** o write path de `IDV`, religando `Disturbance` (já implementado) do mesmo jeito. Esses quatro itens não dependem de nenhum trabalho de física pendente — são fiação, não implementação nova.

`XMEAS` completo e o flag de shutdown ficam necessariamente para depois, porque dependem de fechar `Flows`/`Heat`/`Measurements` (os três `todo!()` que travam o pipeline de química) — não há atalho que os desacople disso. Controladores (PID) não deveriam entrar no escopo mínimo de `AcquisitionLayer`: por definição eles são consumidores de sinais (leem XMEAS, escrevem XMV), não uma fonte de sinal, então cabem numa camada acima do gateway, não dentro dele. Em resumo: o gateway OPC-UA tem, hoje, zero sinais realmente prontos para expor fora dos valores físicos brutos já calculados por `Reactor`/`Separator`/`Stripper`/`Compressor` — e mesmo esses exigem escrever a primeira função de snapshot, porque não existe getter nenhum ainda.
